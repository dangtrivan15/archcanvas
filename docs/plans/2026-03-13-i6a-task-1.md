## Description

This task creates the foundation that all other I6a tasks depend on: the shared type definitions, the system prompt builder, and the mock Claude Code SDK for testing.

### `src/core/ai/types.ts`

Define all shared types for the AI integration layer:

- **`ChatEvent`** — discriminated union with 7 variants: `text`, `tool_call`, `tool_result`, `thinking`, `permission_request`, `done`, `error`. Every variant carries a `requestId: string` field for WebSocket stream correlation.
- **`ClientMessage`** — discriminated union with 4 variants: `chat` (with `requestId`), `abort`, `load_history`, `permission_response`.
- **`ChatMessage`** — `{ role: 'user' | 'assistant', content: string, events?: ChatEvent[], timestamp: number }`.
- **`ProjectContext`** — `{ projectName, projectDescription?, currentScope, projectPath }`.
- **`ChatProvider`** interface — `{ id, displayName, available, sendMessage(), loadHistory(), abort() }`. `sendMessage` returns `AsyncIterable<ChatEvent>`.

Use plain TypeScript types (no Zod here — these are internal interfaces, not user-facing validation boundaries).

### `src/core/ai/systemPrompt.ts`

A pure function `buildSystemPrompt(context: ProjectContext): string` that constructs the system prompt injected into the Claude Code session. The prompt must include:

- Role description ("architecture assistant for ArchCanvas")
- Project path
- All 9 CLI command signatures with correct flags (especially: `remove-edge --from/--to`, `add-edge --from/--to/--fromPort/--toPort`)
- Current scope and project name
- Instruction to always use `--json` flag

Test: verify the output string contains all command signatures and interpolated context values.

### `test/mocks/mockClaudeCode.ts`

A mock implementation that emits realistic Claude Code SDK event sequences. Define 7 predefined scenarios as named exports:

1. **`textStreaming`** — `text → text → text → done`
2. **`toolCallFlow`** — `text → permission_request → (approved) → tool_call → tool_result → text → done`
3. **`permissionDenied`** — `text → permission_request → (denied) → text → done`
4. **`clarifyingQuestion`** — `text (question) → done`
5. **`errorScenario`** — `text → error`
6. **`abortMidStream`** — `text → text → (abort signal) → done`
7. **`multipleMutations`** — `tool_call → tool_result → tool_call → tool_result → text → done`

Each scenario is an async generator function that yields `ChatEvent` objects with realistic delays (configurable, defaulting to 0 for tests). The mock should accept a `requestId` parameter and stamp it on all emitted events.

### Acceptance Criteria
