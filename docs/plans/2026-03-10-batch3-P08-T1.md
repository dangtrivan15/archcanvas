## Description

This task defines the storage abstraction layer that all backends implement, plus
a StorageManager that composes backends with the codec and proto conversion pipeline.

### StorageBackend Interface

```typescript
// src/core/storage/types.ts

export interface StorageBackend {
  readonly type: string;

  // Core I/O — read/write raw bytes
  read(handle: StorageHandle): Promise<Uint8Array>;
  write(handle: StorageHandle, data: Uint8Array): Promise<StorageHandle>;

  // File picker operations
  openFilePicker(options?: OpenPickerOptions): Promise<StorageHandle | null>;
  saveFilePicker(data: Uint8Array, options?: SavePickerOptions): Promise<StorageHandle | null>;

  // Capabilities
  readonly capabilities: StorageCapabilities;
}

export interface StorageCapabilities {
  supportsDirectWrite: boolean;    // Save in place (vs download)
  supportsLastModified: boolean;   // Can track file modification time
}

export interface StorageHandle {
  readonly backend: string;        // Backend type identifier
  readonly name: string;           // Display name (filename)
  readonly _internal: unknown;     // Backend-specific data (FileSystemFileHandle, File, etc.)
}

export interface OpenPickerOptions {
  accept?: string[];               // File extensions: ['.archc']
}

export interface SavePickerOptions {
  suggestedName?: string;
}
```

### Design Decisions

**Why not extend FileSystemAdapter?** The existing `FileSystemAdapter` (from P06) is
a platform adapter — it abstracts browser vs. native vs. Node.js. `StorageBackend`
is a storage adapter — it abstracts local vs. cloud vs. in-memory. They operate at
different levels:

```
fileStore → StorageManager → StorageBackend → FileSystemAdapter (for local backends)
                           → codec (encode/decode)
                           → protoConversion (graph ↔ proto)
```

In P08-T2, the `FileSystemAccessBackend` will internally use `webFileSystemAdapter`
functions. The `FileDownloadBackend` will use blob download (currently inline in
webFileSystemAdapter). This layering preserves P06's platform abstraction.

**StorageHandle vs existing `unknown` handle:** fileStore currently stores
`fileHandle: unknown`. StorageHandle adds a typed wrapper with `backend` identifier
and `name` — this allows the StorageManager to route operations to the correct backend.
The `_internal` field holds the actual backend-specific data (FileSystemFileHandle,
File object, path string, etc.).

### StorageManager

```typescript
// src/core/storage/storageManager.ts

export class StorageManager {
  private backend: StorageBackend;

  constructor(backend: StorageBackend) {
    this.backend = backend;
  }

  get backendType(): string { return this.backend.type; }
  get capabilities(): StorageCapabilities { return this.backend.capabilities; }

  // Open: pick file → read bytes → decode binary → convert proto → return graph
  async openArchitecture(options?: OpenPickerOptions): Promise<OpenResult | null>;

  // Save: convert graph → encode binary → write bytes
  async saveArchitecture(graph: ArchGraph, handle: StorageHandle, options?: SaveOptions): Promise<StorageHandle>;

  // Save As: convert → encode → pick location → write
  async saveArchitectureAs(graph: ArchGraph, suggestedName: string): Promise<SaveAsResult | null>;

  // Read raw: pick → read → decode (no proto conversion — for advanced use)
  async readRawFile(handle: StorageHandle): Promise<DecodedFile>;
}
```

The StorageManager uses:
- `encode()` / `decode()` from `codec.ts` for binary format
- `protoToGraphFull()` / `graphToProto()` from `fileIO.ts` for proto conversion
- Backend methods for actual I/O
