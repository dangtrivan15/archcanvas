## Description

This task adds two independent enhancements:

### 1. AskUserQuestion Option Previews

The SDK's `toolConfig.askUserQuestion.previewFormat` setting adds a `preview` field to each question option — containing either markdown or HTML that shows a visual mockup of the choice.

**claudeCodeBridge.ts:** Add `toolConfig: { askUserQuestion: { previewFormat: 'markdown' } }` to the SDK options. We choose markdown because it is safe to render as plain text (or with a lightweight markdown renderer) without XSS concerns. No raw HTML injection is needed.

**ChatQuestionCard.tsx:** For each option that has a `preview` field:

1. Render the preview below the option's description text.
2. Since we chose `'markdown'` format, render the preview inside a `<pre>` block with `whitespace-pre-wrap` styling. The previews typically contain ASCII art and fenced code blocks. Plain text rendering is sufficient and avoids any XSS risk.
3. Wrap the preview in a bordered container with `max-height: 200px` and `overflow-y: auto` to prevent excessively tall previews from breaking layout.
4. If `preview` is undefined (Claude chose not to generate one for that option), render nothing — just the label and description as before.

### 2. PreToolUse Auto-Approve Hooks

Currently, every tool call goes through `canUseTool`, which sends a permission_request to the browser and blocks until the user responds. For read-only tools (`Read`, `Glob`, `Grep`), this is unnecessary friction — they can't modify anything.

**claudeCodeBridge.ts:** Add a `PreToolUse` hook in the SDK options that auto-approves read-only tools:

```typescript
hooks: {
  PreToolUse: [{
    matcher: 'Read|Glob|Grep',
    hooks: [async (input, toolUseID, { signal }) => ({
      hookSpecificOutput: {
        hookEventName: 'PreToolUse',
        permissionDecision: 'allow',
        permissionDecisionReason: 'Read-only tool auto-approved',
      }
    })]
  }]
}
```

This hook runs before `canUseTool` in the SDK's permission evaluation chain. When it returns `permissionDecision: 'allow'`, the SDK skips `canUseTool` entirely for these tools. The user never sees a permission card for Read/Glob/Grep.

Per SDK docs, hooks execute FIRST in the evaluation chain. `plan` mode prevents all tool execution at the SDK level (before hooks run), so there's no conflict with the auto-approve hook. In `dontAsk` mode, we still want Read/Glob/Grep to work, so the hook is beneficial there too.

### Acceptance Criteria

- `toolConfig.askUserQuestion.previewFormat` is set to `'markdown'` in SDK options
- Question cards render option previews when the `preview` field is present
- Previews render as plain text inside a `<pre>` block (no raw HTML injection)
- Preview containers have max-height to prevent layout breakage
- Options without previews render normally (label + description only)
- Read, Glob, Grep are auto-approved without showing permission cards
- Auto-approve hook does not interfere with other permission modes
- Bash, Write, Edit still trigger permission prompts as before
- New test cases verify the hook returns the correct output for read-only tools
