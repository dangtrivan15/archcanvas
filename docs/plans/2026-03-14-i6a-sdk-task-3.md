## Description

Task 1 enables `includePartialMessages: true` and adds the new event types to our type system. This task handles the actual translation and rendering.

### claudeCodeBridge.ts â€” translateSDKStream expansion

Currently we handle 4 SDK message types: `system`, `assistant`, `user`, `result`. The SDK emits many more. Add handling for:

1. **`stream_event`** (SDKPartialAssistantMessage) â€” These contain `BetaRawMessageStreamEvent` from the Anthropic SDK. Extract `content_block_delta` events that contain text deltas and yield them as `TextEvent`s. This gives us real-time streaming text instead of waiting for the full assistant message. The key is to yield incremental text deltas for `content_block_delta` with type `text_delta`, and thinking deltas for type `thinking_delta`.

2. **`status`** (SDKStatusMessage) â€” Contains a `message` field with status text (e.g., "Reading file...", "Running command..."). Yield as `StatusEvent`.

3. **`tool_progress`** (SDKToolProgressMessage) â€” Live output from running tools (e.g., streaming bash output). Yield as `StatusEvent` with the progress content.

4. **`rate_limit`** (SDKRateLimitEvent) â€” Rate limit warnings. Yield as `RateLimitEvent`.

5. **`prompt_suggestion`** (SDKPromptSuggestionMessage) â€” Suggested next prompts. We can capture these and surface them in the UI as quick-action buttons, but this is lower priority. For now, just capture them in the event stream.

**Important:** When `includePartialMessages` is true, the SDK emits both `stream_event` messages (incremental) AND the final `assistant` message (complete). We need to avoid double-counting text. The approach: when we see `stream_event` deltas, yield them as text events. When the final `assistant` message arrives, skip re-emitting its text content (since it was already streamed), but still process tool_use blocks from it.

### chatStore.ts changes
