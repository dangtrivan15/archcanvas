## Description

E2E tests verifying the persistence UI behavior in a real browser. These test the user-visible behavior, not store internals.

### Test Cases (from spec section 7.8)

| Test | Asserts |
|---|---|
| Adding a node shows dirty indicator in status bar | C8.1, C8.4 |
| Window title shows dirty marker ("●") when modified | C8.2 |
| Cmd+S triggers save (dirty indicator clears) | C10.1, C8.3 |
| File → Save menu item works | C7.2 |
| File → Open Recent submenu shows entries | C7.5 |
| Clean state shows no dirty indicator | C8.3 |

### Testing Strategy

The app runs via `vite preview` (production build on port 4173), same as existing E2E tests.

**Dirty indicator tests:**
1. Load app (clean state) → verify no dirty indicator
2. Add a node via command palette (existing pattern from canvas-operations.spec.ts)
3. Verify dirty indicator appears in status bar
4. Verify window title has "●" prefix

**Save tests:**
- Save-on-close (C9) is hard to test with `beforeunload` limitations
- Instead: verify `beforeunload` handler is registered (via `page.evaluate`)
- Test Cmd+S via `page.keyboard.press('Meta+s')` → verify dirty indicator clears
- Note: actual file I/O in E2E requires the app to have a loaded project with a real FileSystem — may need to mock or use a bootstrapped project

**Menu tests:**
- Open File menu → verify Save item exists with shortcut hint
- Open File menu → verify Save As item exists
- Open File menu → verify Open Recent submenu exists

### Limitations

- `beforeunload` dialog testing is limited in Playwright (spec notes this)
- Web FilePicker (`showDirectoryPicker()`) cannot be automated in Playwright — menu Open/SaveAs would trigger native OS dialogs
- Focus on observable UI state changes (indicator, title) rather than actual file I/O

**Acceptance criteria:**
- Dirty indicator appears/disappears reactively
- Window title updates with dirty state
- Menu items are present and wired
- Keyboard shortcut Cmd+S clears dirty state
- Tests are stable and pass consistently (no flakiness from timing)
