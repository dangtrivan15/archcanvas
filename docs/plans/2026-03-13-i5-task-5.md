## Description

All four mutation commands follow the same pattern:

```
1. loadContext(--project)
2. Validate inputs (e.g., check node type exists in registry)
3. Call graphStore method
4. Check result.ok
5. If ok: fileStore.saveAll(ctx.fs) → format success output
6. If not ok: throw CLIError with engine error code
```

### add-node (Contract C5b)

Flags: `--id` (required), `--type` (required), `--name` (optional), `--scope` (optional), `--args` (optional JSON string)

Key logic:
- Validate `--type` exists in registry BEFORE calling engine (C5b.4: `UNKNOWN_NODE_TYPE`)
- If `--name` omitted, resolve from `registryStore.resolve(type).metadata.displayName` (C5b.5)
- Parse `--args` as JSON if provided
- Construct `InlineNode` object per spec
- Call `graphStore.addNode(resolveCanvasId(scope), node)`

### add-edge (Contract C5c)

Flags: `--from` (required), `--to` (required), `--from-port`, `--to-port`, `--protocol`, `--label`, `--scope`

Construct `Edge` with `EdgeEndpoint` objects: `{ node: from, port: fromPort }`.

### remove-node (Contract C5d)

Flags: `--id` (required), `--scope`

Simple: `graphStore.removeNode(canvasId, id)`. Engine handles cascading edge removal (C5d.4).

### remove-edge (Contract C5e)

Flags: `--from` (required), `--to` (required), `--scope`

Simple: `graphStore.removeEdge(canvasId, from, to)`.

### Testing Strategy

Each command tested via calling the handler function directly (not through commander parse). Common setup:
1. Create InMemoryFileSystem with `.archcanvas/main.yaml`
2. Call `loadContext` (or mock the context setup)
3. Execute command
4. Assert store state and output

Test cases per command follow the test contracts in spec section 7.5.
