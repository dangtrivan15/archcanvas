## Description

This is the foundation task. It expands our type system and bridge plumbing to support all the SDK features that the UI tasks (T2‚ÄìT5) will consume.

### types.ts changes

1. **New PermissionRequestEvent fields:** Add `blockedPath?: string`, `decisionReason?: string` to the existing `PermissionRequestEvent` interface.

2. **New ChatEvent variant ‚Äî `StatusEvent`:** `{ type: 'status'; requestId; message: string }` for SDK status messages and tool progress. Translates from `SDKStatusMessage` and `SDKToolProgressMessage`.

3. **New ChatEvent variant ‚Äî `RateLimitEvent`:** `{ type: 'rate_limit'; requestId; message: string }` for rate limit warnings.

4. **New client message ‚Äî `SetPermissionModeClientMessage`:** `{ type: 'set_permission_mode'; mode: string }` so the browser can change the permission mode mid-session.

5. **New client message ‚Äî `SetEffortClientMessage`:** `{ type: 'set_effort'; effort: string }` so the browser can change effort level.

6. **Expand `PermissionResponseClientMessage`:** Add optional `updatedPermissions?: Array<{ tool: string; permission: 'allow' }>` for the "Always allow" flow.

7. **Expand `PermissionRequestEvent`:** Add optional `interrupt?: boolean` field for deny-with-interrupt.

### claudeCodeBridge.ts changes

1. **Expand `allowedTools`:** Add `'Write'`, `'Edit'`, `'WebFetch'`, `'WebSearch'` to the existing array.

2. **Add `maxTurns: 50`** to the SDK options (safety guard against runaway loops).

3. **Expand `SDKQueryFn` type:** Add `suggestions?`, `blockedPath?`, `decisionReason?`, `agentID?` to the canUseTool options parameter. Add `includePartialMessages?`, `maxTurns?`, `effort?`, `toolConfig?`, `hooks?` to the options object.

4. **Forward `blockedPath` and `decisionReason`** from canUseTool's options to the `onPermissionRequest` callback so the vitePlugin can relay them to the browser.

5. **Forward `interrupt`** from respondToPermission: when denying with interrupt=true, return `{ behavior: 'deny', message: '...', interrupt: true }`.

6. **Forward `updatedPermissions`** from respondToPermission: when allowing with updatedPermissions, return `{ behavior: 'allow', updatedPermissions: [...] }`.

7. **Pass `includePartialMessages: true`** in SDK options so T3 can receive streaming chunks.

8. **Pass `effort`** from a configurable session-level setting (default: `'high'`).

9. **Pass `permissionMode`** from a configurable session-level setting (default: `'default'`), with a method to change it mid-session.

10. **systemPrompt**: Keep our custom prompt but pass it as a plain string (no preset) ‚Äî the preset would override our ArchCanvas-specific context. However, add a comment noting the preset option for future consideration.

11. **Add `hooks` option** with a `PreToolUse` hook that auto-approves `Read`, `Glob`, `Grep` (returns `permissionDecision: 'allow'`). This is configured in `createBridgeSession`, not exposed to the browser.

### vitePlugin.ts changes

1. **Expand `OnPermissionRequest`** callback type to include `blockedPath?` and `decisionReason?`.
2. **Handle new client messages:** `set_permission_mode` and `set_effort` ‚Äî call bridge session methods.
3. **Relay interrupt flag** in permission_response handling.

### webSocketProvider.ts changes

1. **Add `sendSetPermissionMode(mode: string)`** method ‚Äî sends `{ type: 'set_permission_mode', mode }`.
2. **Add `sendSetEffort(effort: string)`** method ‚Äî sends `{ type: 'set_effort', effort }`.
3. **Handle new event types** in `handleMessage` ‚Äî route `status` and `rate_limit` events to the event stream.

### Acceptance Criteria

- `allowedTools` includes Write, Edit, WebFetch, WebSearch, AskUserQuestion
- `maxTurns: 50` prevents runaway loops
- canUseTool receives and forwards blockedPath, decisionReason to permission events
- Permission deny can include interrupt flag
- Permission allow can include updatedPermissions
- includePartialMessages is enabled
- Permission mode and effort are configurable at session level
- PreToolUse hook auto-approves Read, Glob, Grep without triggering canUseTool
- All new types compile cleanly
- Existing tests still pass; new tests cover the expanded bridge options
