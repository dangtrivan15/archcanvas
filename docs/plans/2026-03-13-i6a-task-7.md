## Description

This task creates integration tests that verify the full pipeline works end-to-end, plus Playwright E2E tests for the chat panel UI. Unit tests for individual components are part of their respective tasks — this task covers the cross-cutting scenarios.

### `test/ai/integration.test.ts`

Tests the full message flow using mock SDK:

**Text streaming flow:**
1. Create mock bridge session → create WebSocket provider (with mock WS) → create chatStore
2. Call `chatStore.sendMessage("hello")`
3. Verify: user message added, isStreaming = true, text events build assistant message, done event sets isStreaming = false

**Permission flow:**
1. Send message that triggers `toolCallFlow` mock scenario
2. Verify: permission_request event surfaces in chatStore messages
3. Call `chatStore.respondToPermission(id, true)`
4. Verify: tool_call and tool_result events follow, conversation completes

**Permission denied flow:**
1. Same setup, but respond with `allowed: false`
2. Verify: Claude Code adjusts approach (text response, done)

**Abort flow:**
1. Send message, abort mid-stream
2. Verify: isStreaming becomes false, partial message preserved

**Error flow:**
1. Send message that triggers error scenario
2. Verify: chatStore.error is set, isStreaming = false

**Store action (HTTP mutation) flow:**
1. Simulate receiving a `store_action` message via WebSocket
2. Verify: graphStore method is called with correct args
3. Verify: fileStore.save() is called
4. Verify: result is sent back via WebSocket

**Bridge timeout:**
1. Simulate HTTP mutation request where browser WebSocket disconnects before responding
2. Verify: bridge returns HTTP 502

**Engine error relay:**
1. Simulate store action where graphStore returns `ok: false`
2. Verify: error is forwarded back through WebSocket → HTTP

**Concurrent HTTP mutations:**
1. Send two simultaneous store_action messages
2. Verify: both are processed (serialized through store) and both results returned

### `test/e2e/ai-chat.spec.ts`

Playwright E2E tests for the chat panel UI:

**Chat panel toggle:**
1. Click AI Chat button in toolbar → verify right panel shows ChatPanel
2. Click again → verify right panel returns to details mode
3. `Cmd+Shift+I` → verify toggle works via keyboard

**Chat panel elements:**
1. Open chat panel → verify header, provider selector, input box, send button visible
2. Verify provider selector shows "Claude Code"
3. Verify input is auto-focused

**Send message (with mock):**
Note: E2E tests run against `vite preview` (production build). The Vite plugin only runs in dev mode. For E2E tests, we need to either:
- Option A: Skip live AI interaction tests (just test UI rendering and interactions)
- Option B: Mock the WebSocket at the Playwright level

**Recommendation: Option A** — test UI components and interactions (toggle, input, button states) without a live bridge. The integration tests (above) cover the full pipeline with mocks.

E2E scenarios:
1. Toggle chat panel open/close
2. Verify panel layout (header, body, footer)
3. Verify keyboard shortcut
4. Verify chat button highlights when active
5. Verify right panel returns to details when chat is closed and a node is selected

### `test/ai/cli-bridge.test.ts`

Integration tests for CLI bridge detection and HTTP client:

**Bridge detection:**
1. Start mock HTTP server → `detectBridge()` returns URL
2. No server → `detectBridge()` returns null
3. Server returns non-200 → `detectBridge()` returns null

**Bridge mutation:**
1. Mock HTTP server returns `{ ok: true, ... }` → `bridgeMutate()` returns result
2. Mock server returns `{ ok: false, error: {...} }` → throws CLIError with correct code/message
3. Mock server returns HTTP 504 (timeout) → throws CLIError
4. Network error → throws CLIError('BRIDGE_ERROR')

**CLI command with bridge:**
1. Run `archcanvas add-node` via `execFile` with mock bridge server running
2. Verify: HTTP POST received by mock server with correct args
3. Verify: CLI outputs the mock server's response as JSON

**CLI command without bridge:**
1. Run `archcanvas add-node` via `execFile` with no bridge running
2. Verify: command falls back to direct file mutation (existing behavior)
