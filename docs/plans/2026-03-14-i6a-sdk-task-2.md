## Description

This task upgrades the permission card UI to take advantage of the richer context the SDK now provides (via T1).

### ChatPermissionCard.tsx changes

1. **Show `blockedPath`** when present — display "File: /path/to/file.ts" below the command, so the user knows which file triggered the permission request.

2. **Show `decisionReason`** when present — display a dimmed explanation line like "Reason: This tool is not in the allow list" so the user understands *why* they're being asked.

3. **"Always Allow" button** — a third button alongside Approve/Deny. When clicked, calls `respondToPermission(id, true, { updatedPermissions: [{ tool, permission: 'allow' }] })`. This tells the SDK to add a permanent allow rule for this tool, preventing future prompts.

4. **"Deny & Stop" button** — replaces the current Deny button with a split: "Deny" (continues agent, just skips this tool) and "Deny & Stop" (sends `interrupt: true`, which halts the agent entirely). Useful when the user wants to abort a dangerous operation chain.

5. **Visual improvement** — use a warning/shield icon or colored left border to distinguish permission cards from regular messages.

### chatStore.ts changes

1. **Expand `respondToPermission` signature** to accept an optional third argument: `options?: { updatedPermissions?: Array<{ tool: string; permission: 'allow' }>; interrupt?: boolean }`.

2. **Forward these options** to the WebSocket provider's `sendPermissionResponse` method (which T1 will have expanded).

### Acceptance Criteria

- Permission card shows file path when `blockedPath` is present
- Permission card shows reason when `decisionReason` is present
- "Always Allow" button sends updatedPermissions in the response
- "Deny & Stop" button sends interrupt: true
- Regular "Deny" button continues to work as before (no interrupt)
- Card is visually distinct with contextual information
- Existing permission flow still works for tools without blockedPath/decisionReason
