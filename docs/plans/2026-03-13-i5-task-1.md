```typescript
// src/core/registry/builtins/compute/service.ts
import type { NodeDef } from '../../../../types/nodeDefSchema';

export const serviceDef: NodeDef = {
  kind: 'NodeDef',
  apiVersion: 'v1',
  metadata: { name: 'service', namespace: 'compute', /* ... */ },
  spec: { /* ... */ },
};
```

## Description

The current builtin NodeDefs are loaded as raw YAML strings via Vite's `?raw` import suffix, then parsed at runtime. This works in the browser (Vite handles `?raw`) but fails in Node.js (the CLI target), since the CLI uses a separate Vite build that doesn't serve `?raw` imports at runtime.

**Solution:** Convert each YAML file to a TS file exporting a plain object literal that satisfies the `NodeDef` type. The barrel `builtins/index.ts` collects all 32 into a `builtinNodeDefs` array. `registryStore.initialize()` uses this array directly — no YAML parsing, no `?raw` imports, no environment detection.

**Approach:**
1. Establish the conversion pattern with 1-2 exemplar files (compute/service, data/database)
2. Convert all remaining files mechanically — each is a 1:1 translation of YAML structure to TS object literal
3. Update namespace barrels to export `NodeDef[]` arrays instead of raw YAML strings
4. Update the main barrel, loader, and registryStore
5. Verify all existing registry tests pass

**Acceptance criteria:**
- `builtinNodeDefs` is a static array of 32 validated `NodeDef` objects (Contract C3.1–C3.5)
- `registryStore.initialize()` uses `builtinNodeDefs` directly (C3.6)
- All existing registry tests pass unchanged (or with minimal import updates)
- No `?raw` imports remain in the builtins loading path
- YAML files may be kept alongside for reference (optional — spec says they're no longer loaded at runtime)

**NOTE:** This task exceeds the 10-file hard cap. The 32 YAML→TS conversions are mechanical data literals, not logic. Implementation should parallelize by namespace (9 independent groups).
