## Description

This task extracts the AI bridge server logic from `vitePlugin.ts` into a standalone, reusable server module. The goal: the same bridge code runs inside Vite dev server, in `vite preview`, and as a standalone process for the Tauri sidecar.

**Bridge server extraction (`bridgeServer.ts`):**

Extract from `vitePlugin.ts`:
1. HTTP route handling — all `POST /__archcanvas_ai/api/*` routes (mutations: add-node, add-edge, remove-node, remove-edge, import; reads: list, describe, search, catalog)
2. WebSocket upgrade handling — the `upgrade` event handler that manages Claude Code SDK sessions
3. Unhandled rejection handler (lines ~110-125 of current vitePlugin) — SDK workaround
4. Session cleanup on WebSocket close

The server should expose a clean API:
```typescript
interface BridgeServerOptions {
  port?: number;      // default 17248
  cwd?: string;       // project working directory
  host?: string;      // default '127.0.0.1'
}

function createBridgeServer(options: BridgeServerOptions): {
  start(): Promise<{ port: number }>;
  stop(): Promise<void>;
}
```

**vitePlugin refactoring:**

After extraction, `vitePlugin.ts` becomes a thin wrapper:
- `configureServer` hook: creates a bridge server instance, attaches its request handler to Vite's server using middleware
- `configurePreviewServer` hook: same pattern (this is the bonus fix for `vite preview`)
- No route logic remains in vitePlugin — it's all in bridgeServer

**Standalone entry (`src/bridge/index.ts`):**

A simple CLI that creates and starts the bridge server:
```typescript
import { createBridgeServer } from '../core/ai/bridgeServer.js';
const server = createBridgeServer({ port: args.port, cwd: args.cwd });
const { port } = await server.start();
console.log(`BRIDGE_PORT=${port}`);  // structured first line for Tauri sidecar
```

**Build config (`vite.config.bridge.ts`):**

Follow the same pattern as `vite.config.cli.ts`: library mode, ESM output, `node20` target, external node_modules, shebang injection. Output: `dist/bridge-server.js`.

**Key risk:** The extraction must not break the existing `vite dev` experience. The vitePlugin refactoring is the most delicate part — ensure the middleware integration preserves the same request/response behavior.

**Acceptance criteria:**
- `node dist/bridge-server.js --port 17248 --cwd /tmp/test` starts a working bridge server
- WebSocket connections to the standalone server work (session create, message exchange)
- HTTP routes respond correctly (mutations and reads)
- `vite dev` still works with the refactored vitePlugin (no behavior change)
- `vite preview` now has bridge support via `configurePreviewServer`
- `npm run build:bridge` produces `dist/bridge-server.js`
- Graceful shutdown on SIGTERM/SIGINT
