## Description

This task modifies the CLI's mutation commands to detect a running bridge and route mutations through HTTP instead of local file writes. Read-only commands (`list`, `describe`, `search`) are unaffected.

### `src/cli/context.ts` Changes

Add two new exports:

**`detectBridge(): Promise<string | null>`**
- Sends `GET http://localhost:5173/__archcanvas_ai/health` with a short timeout (2 seconds)
- Returns the base URL if successful, `null` if connection fails
- Uses Node.js `fetch` (available in Node 18+)
- Add a comment explaining the `strictPort: true` dependency in `vite.config.ts`

**`bridgeMutate(bridgeUrl: string, action: string, args: Record<string, unknown>): Promise<{ ok: boolean; [key: string]: unknown }>`**
- Sends `POST {bridgeUrl}/api/{action}` with JSON body `args`
- Returns the parsed JSON response
- On HTTP error (non-2xx): throws `CLIError` with the error code/message from the response body
- On network error: throws `CLIError('BRIDGE_ERROR', message)`

Update `CLIContext` interface to include `bridgeUrl: string | null`. Set it during `loadContext()` by calling `detectBridge()`.

### Mutation Command Changes

Each mutation command gets the same pattern change. Before (current):

```typescript
const result = useGraphStore.getState().addNode(canvasId, node)
if (!result.ok) throw new CLIError(...)
await useFileStore.getState().saveAll(ctx.fs)
printSuccess(...)
```

After:

```typescript
if (ctx.bridgeUrl) {
  const result = await bridgeMutate(ctx.bridgeUrl, 'add-node', { canvasId, node })
  if (!result.ok) throw new CLIError(...)
  printSuccess(result)
} else {
  // existing local path unchanged
  const result = useGraphStore.getState().addNode(canvasId, node)
  if (!result.ok) throw new CLIError(...)
  await useFileStore.getState().saveAll(ctx.fs)
  printSuccess(...)
}
```

The bridge path skips `loadContext()` store initialization for mutations (the browser stores are the authority). However, `loadContext()` still runs to resolve project root and detect the bridge — it just skips the expensive store initialization when `bridgeUrl` is set.

Wait — actually, `loadContext()` must still initialize stores for the fallback path and for argument validation (e.g., checking that `--type` exists in the registry). The optimization to skip store init when bridge is detected is a future improvement. For I6a, `loadContext()` runs fully regardless, and the bridge path is just an alternative output path.

### Import Command

`import.ts` has a different structure — it takes `ctx` as an argument (not loaded internally) and does bulk operations with error collection. The bridge path for import sends a single `POST /__archcanvas_ai/api/import` with the full YAML content as the body. The bridge handler parses and applies all operations, returning counts + errors.

### Acceptance Criteria

- `detectBridge()` returns URL when bridge is running, null when not
- `bridgeMutate()` sends correct HTTP requests and parses responses
- All 5 mutation commands use bridge path when available, fallback when not

- Bridge errors (timeout, disconnect) surface as proper CLIError with code/message
