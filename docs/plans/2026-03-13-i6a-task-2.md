## Description

This task builds the server-side infrastructure: the Claude Code SDK wrapper and the Vite plugin that hosts it. These are Node.js-only modules that must never be bundled into the browser build.

### `src/core/ai/claudeCodeBridge.ts`

Wraps the `@anthropic-ai/claude-code` SDK to provide session management:

```typescript
interface BridgeSession {
  sendMessage(content: string, context: ProjectContext): AsyncIterable<ChatEvent>
  respondToPermission(id: string, allowed: boolean): void
  loadHistory(messages: ChatMessage[]): void
  abort(): void
  destroy(): void
}

function createBridgeSession(options: { cwd: string }): BridgeSession
```

Implementation details:
- `createBridgeSession` initializes a Claude Code SDK session with the given `cwd`
- `sendMessage` calls `buildSystemPrompt(context)` to construct the system prompt, then calls the SDK with the user message. Translates SDK events into `ChatEvent` objects (stamping `requestId`).
- Permission handling: when the SDK emits a permission request, the bridge emits a `permission_request` ChatEvent and pauses. `respondToPermission` resumes the SDK with the user's decision.
- `abort` cancels the current SDK operation.
- `destroy` cleans up the SDK session.
- All SDK-specific types are internal â€” the public interface uses only types from `types.ts`.

Tests: Use the mock SDK (from Task 1) instead of the real `@anthropic-ai/claude-code`. Test all 7 mock scenarios: text streaming, tool calls, permission approve/deny, clarifying questions, errors, abort, multiple mutations.

### `src/core/ai/vitePlugin.ts`

A Vite plugin that hosts both WebSocket and HTTP endpoints on the dev server:

**WebSocket endpoint** (`ws://localhost:5173/__archcanvas_ai`):
- On connection: create a `BridgeSession` per WebSocket client
- Handle `ClientMessage` types: `chat` â†’ call `session.sendMessage()` and stream ChatEvents back; `abort` â†’ call `session.abort()`; `load_history` â†’ call `session.loadHistory()`; `permission_response` â†’ call `session.respondToPermission()`
- On disconnect: call `session.destroy()`

**HTTP mutation endpoints** (on `server.middlewares`):
- `GET /__archcanvas_ai/health` â†’ `{ ok: true }`
- `POST /__archcanvas_ai/api/add-node` â†’ relay to browser via WebSocket `store_action` message, wait for result, return as HTTP response
- `POST /__archcanvas_ai/api/add-edge` â†’ same pattern
- `POST /__archcanvas_ai/api/remove-node` â†’ same pattern
- `POST /__archcanvas_ai/api/remove-edge` â†’ same pattern
- `POST /__archcanvas_ai/api/import` â†’ same pattern

**HTTP mutation handling:**
- Receive mutation args from CLI via HTTP POST
- Send `{ type: 'store_action', action: 'addNode', args: {...}, correlationId }` via WebSocket to browser
- Wait for browser response with matching `correlationId` (timeout: 10 seconds)
- Return result as HTTP JSON response
- On timeout: HTTP 504 with `{ ok: false, error: { code: 'BRIDGE_TIMEOUT' } }`
- On WebSocket disconnect: HTTP 502 with `{ ok: false, error: { code: 'BRIDGE_DISCONNECTED' } }`

**Browser bundling prevention:**
- Add `claudeCodeBridge.ts` and `vitePlugin.ts` patterns to `build.rollupOptions.external` in `vite.config.ts`
- These files import `@anthropic-ai/claude-code` and `ws` (Node.js only) â€” must never enter the browser bundle
- Follow the same pattern used for Tauri API externalization

**Vite config changes:**
- Register `aiBridgePlugin()` in the plugins array
- Add `@anthropic-ai/claude-code` and `ws` to rollup externals
- Add comment on `strictPort: true` explaining bridge detection dependency

Tests: WebSocket lifecycle (connect/disconnect/reconnect), HTTP health endpoint, HTTP mutation timeout, HTTP mutation disconnect, permission relay roundtrip.
