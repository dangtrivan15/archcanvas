## Description

This task creates the foundational dialog registry infrastructure that all subsequent
P01 tasks build on.

### Architecture

The dialog registry replaces the pattern of hardcoding each dialog in App.tsx with a
declarative system:

1. **`types.ts`** — Defines `DialogConfig` (id + component + optional defaults) and a
   `DialogId` string literal union type for all known dialog IDs. Also defines a
   `DialogPropsMap` type that maps each dialog ID to its specific props type (the
   existing info interfaces like `DeleteDialogInfo`, `ConnectionDialogInfo`, etc.).

2. **`registry.ts`** — A `Map<string, DialogConfig>` with `registerDialog()` and
   `getDialog()`. Dialogs self-register at import time. No dynamic loading needed.

3. **`DialogHost.tsx`** — Reads `openDialogs` from uiStore, iterates the Set, looks
   up each dialog's component in the registry, renders it with props from `dialogProps`.
   This single component replaces all 14 inline dialog conditionals in App.tsx.

4. **`uiStore.ts` changes** — Add two new state fields:
   - `openDialogs: Set<string>` — Set of currently open dialog IDs
   - `dialogProps: Map<string, unknown>` — Props for each open dialog

   Add two new generic actions:
   - `openDialog(id: string, props?: unknown)` — Adds to Set + Map
   - `closeDialog(id: string)` — Removes from Set + Map

   **Critical: Backwards compatibility.** The existing `openDeleteDialog(info)` etc.
   actions must continue working. Refactor them to internally call
   `openDialog('delete', info)`. This means P01-T2/T3 (dialog migration) can proceed
   incrementally without breaking callers — the ~40 files that call `openDeleteDialog()`
   don't need immediate changes.

### Key Decisions

- Dialog IDs are string literals, not enums (extensible, no registry import needed by callers)
- Existing dialog info interfaces (DeleteDialogInfo, etc.) are preserved — they become
  the props type for each dialog
- `openDialog`/`closeDialog` are synchronous Zustand actions (same pattern as existing)
- No event bus involvement — dialog state is pure Zustand store state

### Acceptance Criteria

- `DialogHost` renders no dialogs when `openDialogs` is empty
- `openDialog('test', { foo: 1 })` + `closeDialog('test')` roundtrips correctly
- Existing `openDeleteDialog(info)` still works (delegates to generic)
- `./scripts/test.sh` passes (existing tests unbroken)
- `npm run build` succeeds
