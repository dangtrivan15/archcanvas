## Description

Final P08 task: wire fileStore to use StorageManager instead of calling fileIO
functions and file adapter directly.

### fileStore Changes

**State field change:**
```typescript
// Before
fileHandle: unknown;

// After
fileHandle: StorageHandle | null;
```

This is the key type improvement — `StorageHandle` carries backend type + name,
replacing the opaque `unknown`.

**StorageManager access:**
```typescript
// Option A: Store a reference in fileStore state
storageManager: StorageManager;

// Option B: Access via engineStore (initialized during app setup)
const { storageManager } = useEngineStore.getState();

// Recommendation: Option B — StorageManager is an engine-level concern, not file state
```

### Action-by-Action Migration

**1. `openFile()` (lines 213-304):**
```typescript
// Before
const picked = await pickArchcFile();
const result = decodeArchcData(picked.data);
// ... integrity error handling ...
_applyDecodedFile(result.graph, picked.fileName, picked.fileHandle, ...);

// After
const manager = useEngineStore.getState().storageManager;
const result = await manager.openArchitecture();
if (!result) return; // cancelled
// ... integrity error handling stays (catch CodecError/IntegrityError) ...
_applyDecodedFile(result.graph, result.handle.name, result.handle, ...);
```

**2. `saveFile()` (lines 306-429):**
```typescript
// Before
await saveArchcFile(graph, fileHandle, ...);
const newLastModified = await getFileLastModified(fileHandle);

// After
const manager = useEngineStore.getState().storageManager;
const updatedHandle = await manager.saveArchitecture(graph, fileHandle, ...);
// lastModified: manager can query from backend if supported
```

**3. `saveFileAs()` (lines 431-511):**
```typescript
// Before

// After
const manager = useEngineStore.getState().storageManager;
const result = await manager.saveArchitectureAs(graph, suggestedName);
```

**4. `loadFromUrl()` (lines 513-597):**
```typescript
// Before

// After: StorageManager doesn't handle URL loading directly.
// Keep the fetch + use manager.decodeRawData() or keep using codec directly.

```

**5. `loadFromDroppedFile()` (lines 599-685):**
```typescript

```

**6. `newFile()` (lines 118-140):**
No storage backend involvement — just creates an empty graph. No changes needed.


### useFilePolling Impact

`useFilePolling.ts` currently reads `fileHandle` as `FileSystemFileHandle` with
duck-typing. With the new `StorageHandle`, it needs to access `handle._internal`
to get the underlying FileSystemFileHandle for `getFile()` calls.

Options:
1. Update useFilePolling to use `(fileHandle as StorageHandle)?._internal`
2. Add a `getLastModified(handle: StorageHandle)` method to StorageBackend
3. Keep using `getFileLastModified()` from fileSystemAdapter (it already duck-types)

**Recommendation:** Option 3 — `getFileLastModified()` already handles unknown types
with duck-typing. Pass `storageHandle._internal` to it.

### Offline Sync (enqueueSave)

The `enqueueSave()` path in `saveFile()` currently calls `encode()` and `graphToProto()`
directly. This stays — the sync queue stores raw encoded bytes, not a backend operation.
The StorageManager is only for interactive save/open paths.
