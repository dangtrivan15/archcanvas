## Description

Migrate the remaining 7 dialogs. These are "complex" because they have non-standard
patterns compared to the simple boolean+info dialogs:

### Special Cases

1. **AnalysisProgressDialog** — Does NOT use uiStore for open state. Instead, it reads
   from `analysisStore` (dialogOpen, progress, error, cancel, closeDialog). The
   `AnalysisProgressDialogConnected` wrapper is defined inline in App.tsx (lines 58-74).

   Migration approach: Move the "Connected" wrapper logic into the dialog component
   itself. Register with id `'analysis-progress'`. The dialog reads from analysisStore
   directly (keep this — it's the right pattern for analysis state). The open/close
   is driven by analysisStore, not the generic uiStore dialog system. Register it
   in the dialog registry so DialogHost can render it, but its open state comes from
   analysisStore.

2. **ExternalAgentDialog** — Similar pattern. `ExternalAgentDialogConnected` wrapper
   defined in App.tsx (lines 79-95). Reads `externalAgentDialogOpen` and
   `externalAgentDialogInfo` from uiStore. Simpler than AnalysisProgress — can fully
   migrate to generic pattern.

3. **ShortcutsHelpPanel / ShortcutSettingsPanel** — These are panels, not modal
   dialogs. They use simple boolean flags (`shortcutsHelpOpen`, `shortcutSettingsOpen`)
   without info objects. Migration is straightforward — register with no-props pattern.

4. **TemplatePicker (386 lines) / TemplateGallery (611 lines)** — Large components
   but standard open/close pattern. TemplatePicker uses `templatePickerOpen`,
   TemplateGallery uses `templateGalleryOpen`. No info objects. Straightforward migration.

5. **SettingsDialog** — Standard boolean flag pattern (`settingsDialogOpen`). Reads
   many settings from uiStore for its form fields, but open/close is simple.

### Acceptance Criteria
